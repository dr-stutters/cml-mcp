# Test Plans library

Formal, human-readable **test plans** for the MCP suite and the validated lab designs —
the *what we test, how, and the pass/fail criteria*, as reviewable documents. The
counterpart to the automated suites (`tests/` in each repo): the code **executes** the
tests, these plans **describe** them and give a customer-presentable structure.

This library is the source that the **customer-facing test report** (roadmap #40) records
results against — every plan ends with an empty [Execution record](#the-40-bridge) table
the report fills in.

> Like [`Custom Designs/`](../Custom%20Designs/), the commands/names/lab IPs here come from
> real validated runs — treat **IPs/ids as lab-specific** (adjust for your environment) and
> the **sequence, cases, and pass criteria** as the reusable part. No credentials live in
> these plans (only `.env` variable names).

## Test levels

Every plan classifies each case by the level that exercises it. The four levels already
exist across the repos:

| Level | What it is | How it runs | Live target? |
|---|---|---|---|
| **Unit** | Mocked HTTP/transport; verifies request-building, parsing, tool logic | `uv run pytest` (in CI: `ruff` + `pytest`) | no |
| **Smoke** | Live read-only pass — confirms the box answers and core reads return data | `uv run python tests/smoke_test.py` | yes (read) |
| **Integration** | Live create → verify → delete round-trips on throwaway objects | `uv run python tests/integration_test.py --write` | yes (write) |
| **Manual/Live** | Driven via the MCP tools / pyATS / packet-tracer / GUI; recorded by hand | operator + agents | yes |

Not every server has all four — FMC/Splunk/WLC have unit + smoke (their write paths are
exercised manual-live via the specialist agents); ISE/Windows/CML add integration/e2e.

## Conventions

- **Test IDs:** `<PREFIX>-NNN`, unique within a plan. Prefixes:
  servers `CML- ISE- FMC- WIN- SPL- WLC-`; designs `NAC- WLNAC- SGT- SDWAN-`.
- **One case per capability/tool-group**, not per tool — the Steps name the tools it
  exercises. (ISE's 184 tools → ~30 capability cases.)
- **Test-case columns:** `ID · Objective · Steps · Expected result / pass criteria · Coverage`.
  The **Coverage** column maps each case to `unit` / `smoke` / `integration --write` /
  `manual-live` and names the tool or test function — this is what makes #40 semi-auto-fillable.
- **Structure:** every plan follows [`_TEMPLATE.md`](_TEMPLATE.md) (8 sections).

## The #40 bridge

Each plan's final **Execution record** table is intentionally empty. The customer-facing
test report (#40) is generated by:

1. Running the automated gate per repo — `uv run pytest`, `smoke_test.py`,
   `integration_test.py --write` — and mapping results onto the `unit` / `smoke` /
   `integration` cases via the Coverage column (pass/fail is machine-readable).
2. Recording the `manual-live` cases from the lab-design proofs (packet-tracer matrices,
   live-session lookups, pyATS pings) with their evidence.
3. Stamping each plan's Execution-record row (date, tester, P/F/skip counts, evidence,
   defects) and rolling them into one report.

## How to add a test plan

1. Copy [`_TEMPLATE.md`](_TEMPLATE.md) to `MCP Servers/<server>.md` or `Lab Designs/<design>.md`.
2. Fill the 8 sections; keep the case-table columns identical.
3. Set the Coverage column from the target's `tests/` (unit fn names / smoke / integration
   round-trip) or `manual-live`.
4. Add a row to the relevant index table above.
